## 1. 簡介

電子燃油噴射（Electronic Fuel Injection, EFI）系統的核心在於**引擎電控單元**（Engine Control Unit, ECU）。ECU 如同引擎的大腦，負責即時接收各種感測器訊號，並透過複雜的演算法，精準控制噴油脈寬（Injector Pulse Width）與點火時機（Ignition Timing），以達到最佳的動力輸出、燃油效率及排放標準。

EFI 系統的出現取代了傳統化油器，使引擎在各種工況下均能維持準確的空燃比（Air-Fuel Ratio, AFR），大幅提升引擎性能與環保性能。近年來，隨著無人機（UAV）活塞引擎應用的興起，車規級 ECU 的設計理念逐漸延伸至航空領域，但兩者在**大氣環境**、**失效策略**及**通訊協議**上存在根本差異，需進行針對性的修改與調適。

## 2. EFI 編程的核心邏輯

### 2.1 燃油方程式

ECU 的首要任務是根據**理想氣體狀態方程式**（Ideal Gas Law, PV = nRT）計算進入氣缸的空氣質量，進而決定噴油脈寬。核心運算公式如下：

```
IPW = K × (MAP × VE / T_air) × λ_target + Accel + Offset
```

| 參數 | 名稱 | 說明 |
|------|------|------|
| `K` | 常數 | 代表噴油嘴流量與排氣量的換算係數 |
| `MAP` | 進氣岐管絕對壓力 | 引擎負載的主要量測依據 |
| `VE` | 容積效率 | 引擎實際進氣量與理論進氣量之比，由 3D 查找表提供 |
| `T_air` | 進氣溫度 | 用於空氣密度修正（溫度越低，空氣越密，需要更多燃油） |
| `λ_target` | 目標空燃比 | 設定的理想燃油混合比例（巡航約 14.7:1，高負載約 12.5:1） |
| `Accel` | 加速補償 | 模擬化油器加速泵，節氣門快速開啟時額外補油 |
| `Offset` | 噴油嘴死區時間 | 補償噴油嘴的物理開啟延遲（通常為數微秒） |

> **UAV 延伸**：在高空飛行時，大氣壓力（BAP）下降導致空氣密度降低，上式中的 MAP 必須以大氣壓力作為基準進行修正，否則供油量將嚴重偏濃。詳見第 7 節。

### 2.2 核心控制查找表（Maps）

EFI 編程的核心之一是建立 **3D 查找表（Look-up Table）**，其 X 軸為引擎轉速（RPM），Y 軸為引擎負載（MAP 或 TPS）：

- **VE 容積效率表**：定義引擎在各轉速與負載組合下的實際進氣效率。
- **點火提前角度表（Ignition Advance Map）**：決定在上死點前（BTDC）幾度觸發火星塞。
- **目標空燃比表（Target AFR Map）**：設定各工況的理想燃油混合目標。
- **Alpha-N 表**（UAV 專用）：以節氣門開度（TPS）為索引的直接供油查找表，在 MAP 感測器延遲或失效時作為備援或混合使用。

### 2.3 關鍵韌體例程

除基本供油外，固件還必須處理多項即時子程序：

- **中斷服務程序（ISR）**：由曲軸位置感測器（CKP）觸發的高優先級程式，確保 ECU 在毫秒甚至微秒級精準掌握活塞位置。
- **冷啟動加濃（Cold Start Enrichment）**：當冷卻水溫或汽缸頭溫度（CHT）偏低時自動加大噴油量。
- **加速補油（Acceleration Enrichment）**：節氣門快速開啟時模擬傳統化油器的加速泵動作。
- **閉迴路修正（Closed-Loop Feedback）**：利用含氧感測器（O2 Sensor）偵測實際燃燒結果，即時修正噴油脈寬。
- **EGT 補濃**（UAV 專用）：排氣溫度（EGT）過高時自動加濃，防止引擎高空熄火或燃燒異常。

## 3. 硬體架構與 MCU 選用

### 3.1 車規級 MCU 需求

車用 ECU 對即時性（Real-time Determinism）與功能安全（Functional Safety）要求極高，通常需要：

- **工作溫度範圍**：−40°C 至 +125°C（部分應用更寬）
- **Lockstep 鎖步核心**：兩個處理器核心同時執行同一段程式，相互比對以即時偵測硬體錯誤
- **GTM（Generic Timer Module）**：專用計時子處理器，處理曲軸觸發輪的齒數計算與噴油嘴觸發，精度達微秒級
- **HSM（Hardware Security Module）**：韌體加密，防止未授權的參數修改
- **ASIL-D 合規**：ISO 26262 最高功能安全等級

### 3.2 主流車用 MCU 比較

| 製造商 | 系列名稱 | 架構特點 | 主要應用場景 |
|--------|----------|----------|-------------|
| Infineon | AURIX™ TC3xx / TC4xx | TriCore™ 多核心架構，強大 DSP 能力，支援 Lockstep | 德系車廠（Bosch、Continental）主流選擇 |
| NXP | S32K / S32G | ARM Cortex-M7 / A53，整合 CAN-FD 與車用乙太網路 | 網域控制器（Domain Controller） |
| ST Micro | Stellar / SPC5 | ARM Cortex-R52，相變記憶體（PCM）支援快速 OTA | 歐系油電混合動力總成 |
| Renesas | RH850 | 32-bit RISC 自研架構，超低功耗、極高可靠性 | 日系車廠（Toyota、Honda、Nissan）首選 |

### 3.3 研究與自製 ECU 的替代方案

對於開源或研究性質的 EFI 開發，常見的替代選擇：

- **STM32 F4 / F7 / H7 系列**：開源 ECU 計畫（如 RusEFI）的最主流選擇，社群資源豐富，最高時脈可達 480 MHz。
- **Teensy（NXP i.MX RT1062）**：600 MHz 的高效能跨界處理器，適合高階 DIY 應用。
- **Raspberry Pi RP2350**：其 **PIO（可程式化 I/O）模組** 能以零 CPU 抖動處理曲軸觸發訊號，功能近似專業 GTM，且雙核 Cortex-M33 架構支援 Core 0（控制迴路）與 Core 1（通訊）的確定性分工，特別適合 UAV 引擎研究應用。

> **建議**：若進行自定義飛行控制器或引擎管理研究，STM32H7 或 RP2350 是兼顧效能與開發成本的優質選擇。

## 4. 感測器融合與卡爾曼濾波器

### 4.1 為何需要感測器融合

車規 ECU 通常直接使用 ADC 的原始讀值（僅作簡單移動平均），因為汽車引擎室的環境相對穩定。然而在 UAV 應用中，以下因素使得原始讀值不可靠：

- 引擎振動引入 ADC 電氣雜訊
- NTC 熱敏電阻本身具有 2–3°C RMS 雜訊
- MAP 感測器在節氣門急開時存在數十毫秒的物理延遲
- 高空氣壓變化需要即時估算空氣密度

**卡爾曼濾波器（Kalman Filter, KF）** 提供了一個最優線性估計框架，能同時融合多個感測器的量測值，輸出雜訊更低、延遲更小的物理量估計。

### 4.2 MAP + BAP 融合卡爾曼濾波器（3 狀態）

車規版採用 2 狀態 KF 估計 `[P_man, dP/dt]`。UAV 版升級為 **3 狀態**，加入大氣壓力狀態：

```
狀態向量：x = [ P_manifold,  dP/dt,  P_baro ]ᵀ
                 進氣岐管壓力  壓力變化率  大氣絕對壓力

狀態轉移矩陣 F：
  ┌ 1  dt  0 ┐
  │ 0   1  0 │     dt = 控制週期 (0.5 ms @ 2 kHz)
  └ 0   0  1 ┘

量測矩陣 H（兩個量測值：MAP ADC + MS5611 BAP）：
  ┌ 1  0  0 ┐    z₁ = P_manifold（ADC 量測）
  └ 0  0  1 ┘    z₂ = P_baro（MS5611 精密量測）
```

**關鍵設計決策**：
- BAP 狀態的過程噪聲 Q 極小（Q₂ ≈ 0.001 kPa²），因為大氣壓力在飛行中變化緩慢（約 0.0012 kPa/s @ 爬升速率 10 m/s）
- MAP 狀態 Q 較大（Q₀ ≈ 0.2 kPa²），允許跟蹤節氣門瞬態
- 當 MAP 感測器失效時，BAP 狀態繼續被 MS5611 更新，保持大氣壓力估計有效

**MAP 相對壓力**（真正的進氣岐管真空度）：
```
P_rel = P_manifold - P_baro    [kPa]
```
此值在不同海拔下均能正確反映引擎負載，是 UAV 引擎標定的核心量。

### 4.3 IAT 卡爾曼濾波器（1 狀態）

```
狀態：x = [ T_intake ]
過程噪聲：Q = 0.01    （進氣溫度變化緩慢）
量測噪聲：R = 4.0     （NTC 雜訊 ~2°C RMS → R = 4）
```

效果：將 NTC 熱敏電阻的原始雜訊從 ~2°C RMS 降低至 < 0.3°C RMS，顯著改善低負載下的供油精度。

### 4.4 EGT 卡爾曼濾波器（1 狀態，UAV 新增）

```
狀態：x = [ T_exhaust ]
過程噪聲：Q = 100.0   （EGT 在急加速時可瞬變 100°C/s → 需要大 Q）
量測噪聲：R = 9.0     （MAX31855 放大器雜訊 ~3°C RMS → R = 9）
```

EGT KF 提供低延遲的排氣溫度估計，用於觸發高空補濃保護邏輯。

### 4.5 感測器融合後的空氣質量估算

融合後的空氣質量計算如下（海拔修正版本）：

```
ρ_air = P_manifold [Pa] / (R_air × T_intake [K])     R_air = 287.05 J/(kg·K)
m_air = ρ_air × V_swept × VE%                        [kg → 轉換為 mg]
```

由於 P_manifold 本身已反映高空引擎的實際進氣壓力，空氣密度降低效應已自然包含於計算中，**無需額外的海拔修正乘數**（備援模式除外）。

## 5. 關鍵感測器與輸入資訊

### 5.1 靜態硬體參數

| 參數 | 說明 |
|------|------|
| 引擎排氣量 | 計算理論進氣質量的基礎（如 60 cc，UAV 引擎） |
| 噴油嘴流量 | 通常以 cc/min 表示，換算燃油脈寬所需 |
| 燃油壓力 | 通常假設固定壓力（如 3 bar），壓力變化時需補正 |
| 曲軸觸發輪型式 | 例如 60-2 齒型：60 齒中缺 2 齒以識別上死點（TDC） |

### 5.2 即時感測器輸入比較（汽車 vs. UAV）

| 感測器 | 汽車版 | UAV 版 | 差異說明 |
|--------|--------|--------|----------|
| 曲軸位置（CKP） | VR 感測器 | 同，增加 EMI 屏蔽 | 點火線圈對 GPS 的干擾需嚴格隔離 |
| 進氣壓力（MAP） | 主要負載指標 | 同，但需 BAP 為基準 | 高空 MAP 讀值需扣除大氣壓才有意義 |
| **大氣壓力（BAP）** | 通常不使用 | **MS5611，精度 ±1.5 Pa** | UAV 核心新增感測器，提供海拔基準 |
| 進氣溫度（IAT） | NTC 熱敏電阻 | 同 | KF 濾波後精度提升 |
| 冷卻水溫（CLT） | 水冷引擎必備 | 替換為汽缸頭溫度（CHT） | UAV 活塞引擎多為氣冷，無冷卻水 |
| **排氣溫度（EGT）** | 偶爾使用 | **MAX31855 + K型熱電偶** | 航空核心指標，觸發高空補濃邏輯 |
| 含氧感測器（O2） | 寬頻 λ 感測器 | 同 | 高空時閉迴路修正暫停於 EGT 警告期間 |
| 節氣門位置（TPS） | 輔助負載指標 | 同，且為 Alpha-N 主索引 | Alpha-N 演算法直接以 TPS 查表供油 |

## 6. 軟體語言與開發規範

### 6.1 主要程式語言

EFI 系統開發高度依賴底層硬體控制，代碼必須具備高度的**執行確定性（Determinism）**。

**C 語言（工業標準）**：允許直接操作硬體暫存器與記憶體位址，編譯產出的二進制程式小且執行迅速。生產級 ECU（Bosch、Denso）與開源計畫（Speeduino、RusEFI）均廣泛使用。

**C++（現代模組化設計）**：物件導向設計使代碼更模組化、易於維護，適合高層監督邏輯。RusEFI 等高效能開源計畫的首選語言，也是 RP2350 UAV-EFI 原型的開發語言。

**組合語言（Assembly）**：通常以「內嵌（Inline）」方式穿插於 C 代碼，用於極限優化的中斷服務程序（ISR）與開機初始化程式，確保在 8000 RPM 下曲軸觸發事件能在微秒內得到零開銷回應。

### 6.2 模型驅動開發（MBD）

在專業車廠，工程師通常以 **MATLAB/Simulink** 建立數學模型，再透過 TargetLink 等工具自動生成符合 **MISRA C 規範**的代碼。MISRA C 禁止危險的編程實踐（如動態記憶體配置），確保代碼永不崩潰或進入無窮迴圈。

### 6.3 各開發環境比較

| 開發環境 | 主要語言 | 最適用場景 |
|----------|----------|-----------|
| 生產級 ECU（Bosch、Denso） | C / MATLAB Simulink | 可靠性、排放合規、量產 |
| 開源計畫（RusEFI） | C++ | 高效能 DIY 硬體、社群協作開發 |
| 入門自製（Speeduino） | C（Arduino 框架） | 四缸引擎入門、操作簡單 |
| UAV 研究（RP2350） | C++ | 卡爾曼融合、Alpha-N、EGT 保護 |
| 演算法研究 | C++ / Python | 強化學習訓練 → 部署至 C++ |

## 7. 車規級 ECU 移植至無人機活塞引擎

### 7.1 移植優勢

車規級零件（符合 AEC-Q100 / ISO 26262）相比一般工業級晶片具有顯著優勢，移植至 UAV 時可直接受益：

- **寬溫工作範圍**：−40°C 至 +125°C，覆蓋高空低溫飛行環境
- **抗振動與 EMC**：汽車引擎室的振動環境與 UAV 引擎震動頻率相近，EMC 設計有效降低對 GPS 與遙控鏈路的干擾
- **成熟的燃油控制演算法**：Bosch / Visteon 體系的 VE 表、閉迴路修正等邏輯可直接移植

### 7.2 核心挑戰與修改方向

#### 7.2.1 大氣壓力與海拔補償（最關鍵）

汽車主要在地面行駛，即便有高山模式，高度變化範圍與速率均遠小於 UAV。

**問題**：無人機在 3000 m 高空時，大氣壓力約 70 kPa（相較於海平面 101.3 kPa），空氣密度下降約 **29%**。若 ECU 仍以海平面大氣壓作為基準計算供油，會導致混合氣過濃，輕則動力下降，重則在高空引擎熄火（Flameout）。

**解決方案**：

新增 **BAP（Barometric Absolute Pressure）感測器**（如 MS5611），透過 ISA 標準大氣的 hypsometric 公式計算飛行高度：

```
H = (T₀/L) × [1 - (P_bap / P₀)^(R·L/g)]
  ≈ 44330 × [1 - (P_bap / 101325)^0.1903]     [公尺]
```

再計算**空氣密度比 σ**，用於備援模式的供油修正：

```
σ = (P_bap / P₀) × (T₀ / T_iat)
```

在 3000 m ISA 標準條件下，σ ≈ 0.74，即空氣密度降低 26%，燃油量應同步減少。

#### 7.2.2 Alpha-N / Speed-Density 自動混合演算法

**Speed-Density 演算法**（以 MAP 為主）在穩態高精度，但在 UAV 快速爬升或節氣門急開時，MAP 感測器的物理響應延遲（數十 ms）會造成瞬態供油誤差。**Alpha-N 演算法**（直接以 TPS 查表）響應快速但精度略低。

本設計採用**動態混合策略**，根據 RPM 範圍與 MAP 變化率（由 KF 的 `dP/dt` 狀態提供）自動切換：

```
blend_factor = max( f(RPM),  f(|dMAP/dt|) )     ∈ [0, 1]

IPW_final = blend × IPW_Alpha-N + (1 - blend) × IPW_Speed-Density
```

| 條件 | blend 值 | 說明 |
|------|----------|------|
| RPM < 2000 或 `\|dP/dt\|` > 20 kPa/s | → 1.0（全 Alpha-N） | 怠速或急加速，MAP 延遲影響最大 |
| RPM > 3500 且穩態 | → 0.0（全 Speed-Density） | 穩定巡航，MAP 準確 |
| 中間狀態 | 線性插值 | 平滑過渡 |

#### 7.2.3 排氣溫度（EGT）保護邏輯

EGT 是航空引擎最重要的燃燒健康指標。高空稀薄空氣條件下，輕微的混合氣偏稀會導致 EGT 急速攀升，並可能造成引擎損壞。

**EGT 三段保護策略**：

| 階段 | EGT 範圍 | ECU 動作 |
|------|----------|----------|
| 正常 | < 750°C | 正常閉迴路控制 |
| 警告（WARN） | 750–850°C | 每超過 50°C 加濃 5%，停止閉迴路修正，向地面站發送告警 |
| 緊急（CRITICAL） | > 850°C | 最大加濃，啟動阻風門電磁閥，地面站緊急告警 |

#### 7.2.4 重量與體積限制

汽車 ECU 通常採用壓鑄鋁殼，重量達 500 g 至 1 kg。UAV 需對此進行以下處理：

- 移除笨重外殼，改用碳纖維或鋁合金薄板散熱片
- 重新設計小型化航空插頭（取代 J1939 等汽車大型接頭）
- 採用衝壓空氣（Ram Air）冷卻取代引擎室自然對流

#### 7.2.5 失效保護策略：Fail-Functional vs. Limp Mode

這是車規級與航空級 ECU **最根本的哲學差異**：

| 項目 | 汽車跛行模式（Limp Mode） | UAV 失效維持（Fail-Functional） |
|------|--------------------------|--------------------------------|
| 設計目標 | 防止引擎損壞，靠邊停車 | **不惜一切維持引擎運轉** |
| MAP 感測器失效 | 固定轉速限制 | 以 BAP × 負載因子替代，繼續供油 |
| IAT 感測器失效 | 警告燈 | 以預設 20°C 繼續計算，標記告警 |
| EGT 緊急 | 可能切斷引擎 | 最大加濃 + 告警，**不切斷引擎** |
| 引擎熄火 | 可接受（靠邊停車） | **不可接受**（直接墜毀） |

> 引擎停止即等同於墜機，因此 UAV-EFI 的核心原則是：**在任何單一感測器失效的情況下，必須使引擎繼續以降級性能（Degraded）模式運行**，而非停機。

### 7.3 通訊協議差異

| 協議 | 汽車 | UAV |
|------|------|-----|
| 主通訊匯流排 | CAN Bus（ISO 11898） | **DroneCAN / UAVCAN v1**（基於 CAN） |
| 診斷協議 | OBD-II / UDS | MAVLink 或自定義 UAVCAN 訊息 |
| 調校工具 | TunerStudio / HP Tuners | 序列 CSV + GCS（地面控制站）整合 |
| 冗餘點火 | 單點火系統（多數） | **雙點火系統**（雙線圈，互相備援） |

### 7.4 汽車 vs. UAV 活塞引擎 ECU 核心對比

| 特性 | 車規級 ECU | UAV 活塞引擎 ECU |
|------|-----------|-----------------|
| 感測器重點 | O2 感測器、爆震感測器 | **BAP、EGT**（新增核心感測器） |
| 空燃比基準 | 固定海平面大氣 | **隨高度動態修正（BAP 基準）** |
| 燃油算法 | 純 Speed-Density | **Speed-Density + Alpha-N 混合** |
| 感測器融合 | 簡單移動平均 | **卡爾曼濾波器（2–3 狀態）** |
| 失效策略 | 跛行模式（Limp） | **失效維持（Fail-Functional）** |
| 通訊協議 | CAN Bus、OBD-II | **DroneCAN / UAVCAN v1** |
| 冷卻方式 | 水冷或引擎室對流 | 衝壓空氣（Ram Air）氣冷 |
| 重量目標 | 無嚴格限制 | < 100 g（含殼體） |

## 8. 通訊協議與通用安全機制

### 8.1 通訊協議

- **CAN Bus**：用於向數位儀表板傳送轉速等資料，或接收循跡控制系統的訊號。
- **CAN-FD**：下一代 CAN 協議，支援更高資料傳輸速率，適用現代多控制器整合架構。
- **DroneCAN / UAVCAN v1**：基於 CAN 的航空無人機標準通訊協議，傳送 RPM、MAP、EGT、故障旗標等引擎狀態至地面站。
- **串口 / USB 調校協議**：允許外部電腦即時修改查找表，無需重新燒錄固件。

### 8.2 安全機制

- **PID 控制迴路**：用於怠速空氣控制（IAC）和增壓控制，維持引擎在負載變化下的穩定性，採用抗積分飽和（Anti-Windup）設計。
- **Seqlock 共享記憶體保護**：雙核 RP2350 架構中，Core 0（控制迴路）與 Core 1（通訊）透過序列鎖（seqlock）搭配 DMB 記憶體屏障保護共享狀態，防止資料撕裂（Torn Read）。
- **車規 ASIL-D 功能安全**（生產級）：硬體即時偵測並修正記憶體位元錯誤，進入「安全狀態」。
- **UAV Fail-Functional**：任何單一感測器失效時，ECU 以預設值繼續運行，並向地面站發送故障旗標，而非停機。

## 9. 結論與進階方向

EFI 編程不僅僅是撰寫程式碼，更是將**物理熱力學原理**轉化為**數位控制邏輯**的跨領域工程實踐。將車規級 ECU 移植至無人機活塞引擎，是一個以相對較低成本獲得「準航空級」硬體可靠性的聰明策略，但需要針對大氣環境、失效哲學與通訊協議進行系統性的重新設計。

基於 RP2350 的 UAV-EFI 原型已實現以下功能：3 狀態 MAP+BAP 卡爾曼融合、EGT 保護邏輯、Alpha-N/Speed-Density 動態混合、Fail-Functional 失效維持，以及雙核確定性分工架構。

未來可進一步探索：

- **強化學習（Reinforcement Learning）整合**：以 RL 演算法進行 VE 表自動調適，取代手動靜態標定，適應不同燃油品質與高度條件。
- **自適應幾何控制（Adaptive Geometric Control）**：提升系統在極端動態工況（急劇爬升、俯衝）下的魯棒性。
- **雙點火冗餘**：仿照航空慣例，實現雙線圈獨立點火系統，作為點火系統的終極備援。
- **飛行資料記錄器整合（FDR Lite）**：將 ECU 輸出同步至機載 SD 卡，供事後診斷與標定分析。
